## 1. The Big Picture

Peliku is like a production studio in your browser. Imagine a video editing app, but instead of dragging clips around a timeline, you describe what you want in plain English and AI creates the video clips for you. The system has three major pieces: (1) a **web interface** (the part you see and interact with in your browser — buttons, forms, video players), (2) a **server** (a Python program running on your computer that handles the heavy lifting — talking to AI, saving files, assembling videos), and (3) **AI services in the cloud** (Google's Gemini API — robots that write scripts, generate images, and create videos when the server asks them to). The web interface and server are tightly integrated — every page you see in your browser is generated by the server, and every button you click sends a request to the server to do something.

This is a **monolithic architecture**, which means the entire application is one single program (the Django server) that does everything. The alternative would be **microservices** — splitting the application into many small, independent programs (one for script generation, one for video generation, one for the UI, etc.), but that creates explosion of complexity: how do they talk to each other? How do you deploy them? How do you debug when something breaks? For a personal tool with one user, a monolith is simpler: everything lives in one codebase, runs on one server, and uses one database. You start it, you use it, you stop it. No orchestration, no service discovery, no distributed system headaches.

The system is built with Python and JavaScript. **Python** runs the server (Django framework), talks to AI (the Gemini API has a Python library called `google-genai`), manages the database (storing your projects and clips), and coordinates all the background work (when video generation takes 3 minutes, the server does not freeze — it hands the job to a background worker and tells your browser "I'll let you know when it is done"). **JavaScript** (specifically jQuery and vanilla JS — plain JavaScript with no fancy frameworks) runs in your browser to make the interface interactive: updating status messages, showing video players when generation finishes, handling drag-and-drop when you reorder clips. The server sends HTML to your browser, and JavaScript enhances it to feel smooth.

## 2. How Information Flows

This section walks through what happens when you use Peliku, step by step, from the moment you click a button to the moment you see the result.

### 2.1 A Typical User Request: "Generate Video for Clip 3"

Let's trace what happens when you click "Generate Video" on Clip 3 in your project workspace.

1. **You click "Generate Video" in the browser.** That button is attached to a JavaScript event listener (a piece of code that waits for clicks). When clicked, jQuery (a JavaScript library that makes common tasks easier) gathers some information: the ID of Clip 3, the script text you wrote, which reference images you selected, and your video quality settings (720p, 1080p, or 4K).

2. **Your browser sends a request to the server.** jQuery uses AJAX (Asynchronous JavaScript and XML — a way for browsers to send requests to servers in the background without reloading the entire page) to send all that information to a specific URL on the Django server, like `https://your-server.com/api/clips/3/generate`. The request is in JSON format, a text-based way of structuring data that both JavaScript and Python understand.

3. **The Django server receives the request.** Django is a Python web framework — think of it as a librarian who knows exactly which piece of code to run when you ask for something. Django looks at the URL (`/api/clips/3/generate`), finds the corresponding **view function** (a Python function that handles that specific request), and runs it. That function does a few things:
   - Validates the data (is the clip ID valid? Do you have a script? Is the video quality setting one of the allowed values?).
   - Retrieves Clip 3's data from the **database** (a file called `db.sqlite3` where all your projects, clips, reference images, and settings are stored).
   - Creates a **background thread** (because video generation takes minutes, not milliseconds — the server cannot make your browser wait that long, so it hands off the job to a separate Python thread that runs in the background).
   - Creates a **task record** in the database to track progress (status: running, started_at: timestamp).
   - Returns a **task ID** (a unique code like `abc123`) to your browser so you can check on progress later.

4. **A background thread picks up the job.** Python's `threading` module spawns a new thread (a lightweight subprocess within the Django server) that does the actual work:
   - Reads the script text, reference images, and settings.
   - Calls the **Veo API** (Google's video generation service — part of the Gemini family). This is an HTTP request sent to Google's servers: "Here is a text prompt and some reference images. Generate a 9:16 portrait video, 8 seconds long, at 1080p resolution." Veo responds with "OK, I am working on it. Check back in a minute."
   - The thread polls Veo's API every few seconds, asking "Is it done yet?" When Veo says "Yes, here is the video," the thread downloads the MP4 file (a video container format — the actual video data in a file) and saves it to the **local file system** (a folder on your computer's hard drive, like `media/videos/clip_3_abc123.mp4`).
   - The thread updates the **database** to say "Clip 3's video is ready. It is stored at `media/videos/clip_3_abc123.mp4`."
   - The thread updates the task record in the database (status: completed, completed_at: timestamp).
   - The thread exits.

5. **Your browser checks for updates.** Remember that task ID (`abc123`) Django gave you? JavaScript has been polling the server every 2 seconds (sending a request to `/api/tasks/abc123/status`), asking "Is this done yet?" Django checks the Tasks table in the database (a simple table with columns: task_id, status, progress, result, error) and responds with either "Still working" (and the frontend shows an elapsed time counter) or "Completed! Here is the video file path." When the browser sees "completed," it does two things:
   - Updates the video player on the page (an HTML5 `<video>` element — the standard way browsers display videos) to show the new video.
   - Displays a toast notification at the bottom of the screen: "Clip 3 generated successfully."

6. **You watch the video.** The video player's `src` attribute (source — where the video file lives) points to a URL like `https://your-server.com/media/videos/clip_3_abc123.mp4`. When you click play, the browser sends another request to Django. Django serves the file from the local file system (reads the MP4 file and sends it to your browser as a stream — small chunks of data so you can start watching before the entire file downloads).

That is the full journey: browser → Django → background thread → AI service → file storage → database → browser. Every feature in Peliku follows a similar pattern: you trigger an action, the server coordinates the work, AI does the heavy lifting, the server stores the result, and the browser updates to show you what happened.

### 2.2 Authentication Flow

Peliku has **no authentication system**. No login, no password, no user accounts. You open the app in your browser (by typing the URL, like `http://localhost:8000` if running locally), and you are immediately on the home page. The browser and server do not track "who you are" because there is only one user — you. This is like using a desktop app: you do not log in to Microsoft Word every time you open it; it just starts. Same here.

If Peliku were a multi-user app (it is not), authentication would work like this: the server checks who you are (by asking for a username and password), creates a **session** (a temporary note in memory saying "browser XYZ belongs to user Alice"), and sends a **token** (a random string like a temporary key) to your browser. Every subsequent request from your browser includes that token so the server knows "this is Alice asking for her projects, not Bob's projects." But since Peliku is a solo tool, all of this is skipped. The server assumes every request comes from the owner and shows everything.

### 2.3 Data Read vs. Data Write

**Reading data** (viewing a list of projects, opening a project workspace, previewing a reel) is fast:
- Your browser requests a page (e.g., the home page).
- Django queries the database: "Give me all projects ordered by last-edited date."
- The database returns the data (a list of project titles, thumbnails, creation dates).
- Django plugs that data into an HTML template (a pre-written HTML page with blanks to fill in, like Mad Libs).
- Django sends the filled-in HTML to your browser.
- Your browser displays it (renders the HTML and CSS as a visual page).
- Total time: tens of milliseconds.

**Writing data** (creating a new project, editing a script, regenerating a video) involves more steps:
- Your browser sends data to Django (form submission or AJAX request).
- Django validates the data (is the title too long? Is the video quality setting valid?).
- Django writes to the database using the ORM (Object-Relational Mapper — a Python library that translates "save this project" into SQL, the language databases speak: `INSERT INTO projects (title, description) VALUES ('My Reel', 'A sunset video')`).
- The database writes the data to its file (`db.sqlite3`) on the hard drive.
- Django responds to your browser: "Saved successfully."
- Your browser updates the UI (shows a confirmation message or redirects to a new page).
- Total time: tens to hundreds of milliseconds.

The key difference: reads are fast lookups; writes modify the database file, which takes slightly longer. Writes also trigger side effects: creating a project generates scripts (via AI), saving a script might trigger video regeneration (if you click the button), and downloading a reel triggers video assembly (via FFmpeg, a command-line video processing tool).

### 2.4 Real-Time & Background Flows

**Background jobs** handle anything that takes more than a few seconds. Video generation can take 11 seconds to 6 minutes depending on resolution and server load. The browser cannot wait that long — the page would freeze. Instead, Django spawns a Python thread (a lightweight subprocess), and the browser polls for updates.

Here is what happens:
1. You trigger a long-running action (generate video, assemble reel).
2. Django creates a task record in the database (Tasks table) with a unique ID and status: "running".
3. Django spawns a Python thread to do the work and returns the task ID to your browser immediately (in milliseconds).
4. The **thread** (a separate execution path within the Django process) runs in the background, performing the AI API call, downloading files, and updating the database.
5. Your browser polls Django every 2 seconds: "Is task `abc123` done yet?" Django checks the Tasks table and responds: "Still working" or "Complete, here is the result."
6. When complete, your browser stops polling and updates the UI with the result.

This is called **asynchronous processing**. From your perspective, you click a button, see a loading spinner, and a few minutes later, the result appears. Behind the scenes, the work happened in the background without blocking anything.

**Real-time updates**: Peliku does not have true real-time updates (like a chat app where messages appear instantly without you clicking anything). Everything is triggered by your actions. But the polling-based status updates (checking every 2 seconds) **feel** real-time because the delay is imperceptible.

## 3. The Frontend — What the User Sees

The frontend is everything you see and interact with in your browser: the home page grid, the project workspace, the video players, the buttons, the forms.

### How Pages Are Rendered

Peliku uses **server-side rendering (SSR)**. Here is what that means:
- When you visit a page (like the home page), your browser sends a request to the Django server.
- Django runs a Python function (called a **view**) that gathers all the data for that page (your list of projects from the database).
- Django plugs that data into an HTML template (a reusable HTML file with placeholders for dynamic content, written in Django's template language — similar to Jinja2 or ERB if you have used those).
- Django sends the complete HTML page to your browser.
- Your browser displays it immediately (no waiting, no loading spinners for the page itself).

The alternative is **client-side rendering (CSR)**, used by React or Vue apps: the server sends a nearly empty HTML page and a big JavaScript file. Your browser runs the JavaScript, which fetches data from an API, then builds the HTML on the fly. CSR feels instant when navigating between pages (no full reload), but the first page load is slower. Peliku uses SSR because pages load fast, SEO does not matter (it is a private tool, not a public website), and the simplicity of server-rendering fits a tool with no complex client-side state.

### How the UI Gets Its Data

Most data is embedded in the HTML that Django sends. For example, the home page includes a complete list of projects in the HTML itself (Django loops through the projects from the database and writes each one as a `<div class="project-card">` element in the HTML template).

For actions that do not reload the page (like generating a video or deleting a project), jQuery uses **AJAX** to send requests to Django in the background and receives JSON responses. JSON (JavaScript Object Notation) is a lightweight text format for sending data: `{ "status": "complete", "videoUrl": "/media/videos/clip_3.mp4" }`. JavaScript parses this and updates the page dynamically (changes the video player's source, shows a success message).

### How Styling Works

All visual design is defined in **CSS** (Cascading Style Sheets — the language that controls colors, fonts, spacing, layout). Peliku uses custom CSS (no framework like Tailwind or Bootstrap). Every page loads a single `main.css` file. When your browser receives the HTML, it also downloads `main.css` and applies the styles to the HTML elements (buttons become styled buttons, the grid becomes a grid, the dark theme takes effect).

The layout uses **Flexbox** and **CSS Grid** (modern CSS tools for arranging elements on the page). Flexbox is for one-dimensional layouts (a row of buttons, a vertical list of clips). CSS Grid is for two-dimensional layouts (the home page's grid of project cards).

### How the App Stays Fast

Peliku uses several techniques to keep the interface responsive:
- **Lazy loading**: Video files are only downloaded when you open the project workspace or click play. The home page does not load every video file, just thumbnails (small images).
- **AJAX for updates**: Generating a video does not reload the entire page — only the video player UI updates.
- **Minimal JavaScript**: No large framework like React (which would bundle megabytes of code). The entire JavaScript codebase is a few kilobytes: jQuery (30KB minified) plus custom scripts.
- **Server-side rendering**: The first page you see appears instantly because the HTML is ready when it arrives. No "loading..." screen while JavaScript builds the page.

## 4. The Backend — What Happens Behind the Scenes

The backend is the Django server — a Python program running on your computer (or a server if deployed remotely) that handles everything you do not see: storing data, talking to AI, assembling videos, validating inputs.

### What the Backend Is Responsible For

- **HTTP server**: Django listens for requests from your browser (when you type a URL or click a button).
- **Routing**: Django looks at the URL (e.g., `/projects/new`) and decides which Python function to run.
- **Business logic**: Django orchestrates complex workflows (when you create a new project, Django saves the project to the database, calls the Gemini API to generate scripts, saves the scripts to the database, and redirects you to the workspace).
- **Database access**: Django reads and writes data (projects, clips, videos) using the **ORM** (Object-Relational Mapper — you write Python code like `Project.objects.all()`, and Django translates it to SQL: `SELECT * FROM projects`).
- **File handling**: Django saves uploaded images and downloaded videos to the file system and serves them back to your browser.
- **API communication**: Django calls the Gemini API (sends HTTP requests to Google's servers with text prompts, reference images, and video generation requests).
- **Background task coordination**: Django enqueues jobs for Celery (the background worker).

### How API Endpoints Are Organized

Django uses a pattern called **Model-View-Controller (MVC)** — though Django calls it **Model-View-Template (MVT)**. Here is the breakdown:
- **Models**: Python classes representing database tables. For example, the `Project` model represents the `projects` table. Each instance of `Project` is one row in the table.
- **Views**: Python functions handling HTTP requests. For example, the `create_project` view handles POST requests to `/projects/new` (runs when you submit the new project form).
- **Templates**: HTML files with placeholders. The `home.html` template is filled with project data and sent to your browser.

API endpoints (for AJAX requests) are organized by resource:
- `/api/projects/` — list all projects or create a new one
- `/api/projects/<id>/` — get, update, or delete a specific project
- `/api/clips/<id>/generate` — generate a video for a specific clip
- `/api/tasks/<id>/status` — check the status of a background job

Each endpoint is mapped to a view function in Django's URL configuration.

### How Business Logic Is Structured

The "rules" of the application (like "you cannot download a reel unless all clips have videos") live in the view functions and model methods. For example:
- The `download_reel` view checks: does every clip in this project have a `video_file_path` in the database? If not, return an error. If yes, enqueue a Celery task to assemble the videos.
- The `Clip` model has a method `can_extend()` that checks: does this clip have a video? Is the video less than 148 seconds long? (The max after 20 extensions.) If yes, allow extension. If no, disable the button in the UI.

Business logic is not duplicated in JavaScript. The browser trusts the server. When a button should be disabled, the server tells the browser (by passing a boolean like `can_extend=False` to the template). The browser does not make those decisions independently.

### How the Backend Talks to the Database

Django's ORM abstracts the database. You never write raw SQL (though you can if needed). Instead, you write Python:
- `Project.objects.create(title="My Reel", description="A sunset video")` — creates a new row in the `projects` table.
- `Project.objects.filter(created_at__gte="2026-01-01")` — retrieves all projects created after January 1, 2026.
- `clip.script_text = "New script"` and `clip.save()` — updates a clip's script in the database.

Behind the scenes, Django translates these to SQL queries (`INSERT`, `SELECT`, `UPDATE`) and sends them to SQLite (the database). SQLite writes the data to a file on your hard drive (`db.sqlite3`).

### How the Backend Talks to Third-Party Services

Django uses the `google-genai` Python library (the official SDK — software development kit — for Gemini). When you trigger script generation, the view function does something like this:

```python
from google import genai

client = genai.Client(api_key=os.environ["GOOGLE_API_KEY"])
response = client.models.generate_content(
    model="gemini-3-flash-preview",
    contents="Write 7 video clip scripts for a sunset reel..."
)
scripts = response.text
```

This sends an HTTP request to Google's servers (over the internet, using HTTPS — encrypted communication). Google's AI reads the prompt and sends back the generated scripts as text. Django parses the response and saves each script to the database.

The same pattern applies for video generation (Veo) and image generation (Nano Banana) — the SDK abstracts the complexity of API requests.

## 5. The Database — Where Everything Lives

The database is where all your work is stored permanently: project titles, clip scripts, video file paths, reference images, settings. Without the database, every time you close Peliku, you would lose everything.

### What Kind of Database Is Used

Peliku uses **SQLite**, a **relational database**. Here is what that means:
- **Relational**: data is organized into **tables** (like spreadsheets). Each table has rows (individual records) and columns (attributes of those records). Tables are related to each other via **foreign keys** (a way to say "this clip belongs to that project").
- **SQLite**: a self-contained, file-based database. The entire database is a single file (`db.sqlite3`) on your computer. No separate server process, no network connections, no configuration. Python's standard library includes SQLite support, so it works everywhere.

Alternatives:
- **NoSQL databases (MongoDB, Firebase)**: Store data as JSON-like documents instead of tables. Good for unstructured data (like blog posts with varying fields). Not needed here because Peliku's data is highly structured (every project has the same fields: title, description, created_at, etc.).
- **Client-server databases (PostgreSQL, MySQL)**: Separate database programs that run as servers. More powerful, handle concurrent writes better, but require installation and configuration. Overkill for a solo tool.

### Major Data Entities and Relationships

Here are the main "things" Peliku stores and how they relate:

- **Projects**: Each project has a title, description, visual style, aspect ratio, clip duration, number of clips, created_at, and updated_at. One project contains many clips.
- **Clips**: Each clip belongs to one project (foreign key: `project_id`). It has a sequence_number (its position in the reel: 1, 2, 3...), script_text (the prompt), video_file_path (where the MP4 is stored), generation_status (idle, generating, completed, failed), generation_method (text-to-video, image-to-video, etc.), and metadata (duration, resolution).
- **Reference Images**: Each reference image belongs to one project. It has a slot_number (1, 2, or 3), image_file_path (where the PNG is stored), and label (a name the user gives it, like "Main character").
- **Hooks**: Each hook belongs to one project (maximum one hook per project). It has script_text, video_file_path, and an enabled boolean (whether to include it in the final reel).
- **Settings**: A single row storing the user's preferences: default_aspect_ratio, default_clip_duration, video_quality, generation_speed.

Relationships in plain language:
- "A Project has many Clips. Each Clip belongs to exactly one Project."
- "A Project has up to 3 Reference Images. Each Reference Image belongs to exactly one Project."
- "A Project has zero or one Hook. Each Hook belongs to exactly one Project."

### How Data Is Protected

Peliku has **no authentication**, so there are no users to differentiate. All data is accessible to whoever opens the app. The database does not enforce access control (there is no "user_id" column to filter by).

At the file system level, the `db.sqlite3` file is readable and writable by the user account running the Django server. If you are the only user of your computer, this is fine. If others use your computer, the database file is not encrypted — someone with access to the file system can open it with any SQLite viewer and see all your projects. This is acceptable for a personal tool. If privacy is a concern, enable full-disk encryption (BitLocker on Windows, FileVault on macOS, LUKS on Linux) at the operating system level.

### Where Files and Media Are Stored

The database stores **metadata** (information about files), not the files themselves. For example:
- The `Clips` table has a column `video_file_path` like `videos/clip_3_abc123.mp4`.
- The actual MP4 file lives on the file system in `media/videos/clip_3_abc123.mp4`.

Why separate? Databases are optimized for small, structured data (text, numbers, dates). Video files are huge (100MB+). Storing them in the database would make the database file enormous and slow down queries. Instead, Django stores files in a `media/` folder and saves the file path in the database. When you need the file, Django reads the path from the database and serves the file from the file system.

## 6. Third-Party Services — The Helpers

Peliku relies on one main third-party service: **Google Gemini API**. This is where all the "magic" happens — the AI that writes scripts, generates images, and creates videos.

### Google Gemini API

**What it does**: Provides three AI capabilities:
1. **Text generation** (Gemini 3 Flash model) — writes the clip scripts. You send a prompt ("Write 7 video clip scripts for a sunset reel with cinematic vibes"), and it returns structured text ("Clip 1: A glowing orange sun dips toward the horizon...").
2. **Image generation** (Nano Banana models) — creates the reference images and transition frames. You send a prompt ("A young woman with red hair in a black leather jacket, photorealistic portrait"), and it returns a PNG image.
3. **Video generation** (Veo 3.1 models) — creates the video clips. You send a text prompt and optionally reference images, and it returns an MP4 video with audio.

**Why use an external service instead of building this ourselves**: AI models require massive computational resources (thousands of GPUs), terabytes of training data, and years of research. Building a video generation model from scratch is impossible for a solo developer. Google has spent billions on Gemini. Paying per API call ($0.10 to a few dollars per request depending on the operation) is infinitely cheaper and faster than building it yourself.

**When it gets involved**: Every time you click a button related to AI:
- "Generate Scripts" (create new project or regenerate all scripts) → Django spawns a thread that calls Gemini text generation.
- "Generate Video" (on a clip) → Django spawns a thread that calls Veo video generation.
- "Generate" (on a reference image slot) → Django spawns a thread that calls Nano Banana image generation.
- "Generate Transition Frame" → Django spawns a thread that calls Nano Banana image generation (twice, for first and last frame).
- "Generate Hook Script" → Django spawns a thread that calls Gemini text generation.
- "Generate Hook Video" → Django spawns a thread that calls Veo video generation.

**What happens if this service goes down**: If the Gemini API is unavailable (server outage, network issue, or you hit a rate limit), the generation request fails. The thread catches the error, updates the task record (status: failed, error: "API unavailable"), and exits. The frontend shows an error message ("Video generation failed: API unavailable. Please try again."). No data is lost — the clip's script is still in the database, and you can retry when the service is back. Peliku cannot function without Gemini because every core feature depends on it. There is no fallback or alternative (by user choice).

### Redis

**What it does**: Stores the task queue and task results for Celery (the background job runner). When Django says "generate a video for Clip 3," it writes a message to Redis: "Job ID abc123: call Veo API with these parameters." Celery reads messages from Redis, processes them, and writes results back to Redis: "Job abc123: complete, video saved at path X."

**Why use an external service instead of building this ourselves**: Redis is an in-memory database — extremely fast at reading and writing small amounts of data (microseconds). Building a task queue from scratch using Django's database would be slower and more error-prone (handling concurrency, preventing duplicate task execution). Redis is battle-tested, free, and designed for this exact use case.

**When it gets involved**: Every background job (video generation, script generation, image generation, reel assembly).

**What happens if this service goes down**: If Redis crashes while a background job is running, the job is lost (Celery cannot write the result back). The frontend's status polling will eventually time out and show an error. The user can retry. This is rare — Redis is extremely stable and runs locally (no network dependency). If Redis is down before Django starts, Django will not start (it requires Redis to initialize the Celery connection). The deployment guide will ensure Redis starts automatically on boot.

## 7. How Everything Connects — The Integration Points

This section describes the boundaries between systems and how they communicate.

### Frontend ↔ Backend (Browser ↔ Django)

**How they talk**: Two methods:
1. **Full page loads (HTTP GET)**: You type a URL or click a link. Your browser sends an HTTP GET request to Django (e.g., `GET /projects/5/workspace`). Django returns HTML. Your browser replaces the entire page with the new HTML.
2. **AJAX (HTTP POST/GET)**: You click a button that does not reload the page (e.g., "Generate Video"). jQuery sends an HTTP POST request in the background with JSON data (`{ "clipId": 3, "method": "text-to-video", "quality": "1080p" }`). Django processes the request and returns a JSON response (`{ "taskId": "abc123" }`). JavaScript parses the JSON and updates the page (shows a loading spinner and starts polling for status).

**What format the data travels in**: JSON for AJAX requests, HTML for page loads.

**How errors at this boundary are handled**:
- If the server crashes, the browser sees a "connection refused" error. The frontend displays: "Server unavailable. Check that Django is running."
- If the server returns a 4xx error (client error, like invalid input), Django sends a JSON response with an error message: `{ "error": "Clip ID 3 does not exist" }`. JavaScript displays this to the user.
- If the server returns a 5xx error (server error, like an unhandled exception), Django's error page appears. In production, this is customized to show a friendly message: "Something went wrong. Please try again."

**What happens if this connection is slow or fails**: If the AJAX request takes more than 10 seconds, jQuery shows a timeout error: "Request timed out. Please check your internet connection and try again." If the connection fails mid-request, jQuery retries once automatically. If the retry fails, the user sees an error and must manually retry.

### Backend ↔ Database (Django ↔ SQLite)

**How they talk**: Django uses the ORM to generate SQL queries. For example, `Clip.objects.get(id=3)` becomes `SELECT * FROM clips WHERE id = 3`. Django sends this query to SQLite. SQLite reads the `db.sqlite3` file, finds the row, and returns the data. Django wraps the result in a Python object (a `Clip` instance).

**What format the data travels in**: SQL queries (text) in one direction, row data (binary format specific to SQLite) in the other direction. Django's ORM abstracts this entirely — you never see the SQL or the binary data; you work with Python objects.

**How errors at this boundary are handled**:
- If the database file is missing, Django crashes on startup with an error: "`db.sqlite3` not found." The user must run `python manage.py migrate` to create the file.
- If a query fails (e.g., trying to delete a project that does not exist), Django raises a Python exception. The view function catches it and returns an error to the frontend.
- If the database file is corrupted, SQLite refuses to read it. Django crashes. The user must restore from a backup or recreate the database (losing all data).

**What happens if this connection is slow or fails**: SQLite is file-based, so there is no network latency. Queries are microseconds to milliseconds. If the hard drive is extremely slow, queries slow down proportionally, but this is rare on modern hardware. If the file is on a network drive, writes can be much slower — not recommended.

### Backend ↔ Third-Party Services (Django ↔ Gemini API)

**How they talk**: Django (or more precisely, a background thread spawned by Django) uses the `google-genai` Python library, which sends HTTP requests to Google's servers over HTTPS (encrypted). For example, generating a video:
1. Django calls `client.models.generate_video(model="veo-3.1-generate-preview", prompt="...", references=[...])`.
2. The SDK constructs an HTTP POST request with the prompt, reference images (as base64-encoded data or URLs), and settings (aspect ratio, duration, resolution).
3. Google's servers receive the request, queue the job, and return an immediate response: `{ "operationId": "xyz789" }`.
4. Django polls Google's servers every 10 seconds: "Is operation xyz789 done?" Google responds with either "In progress" or "Completed, here is the video URL."
5. Django downloads the video file from the URL and saves it to the file system.

**What format the data travels in**: JSON for all API requests and responses. Images and videos are transferred as binary data or URLs.

**How errors at this boundary are handled**:
- If the API key is invalid, Google returns a 401 Unauthorized error. Django raises an exception, the Celery task fails, and the user sees: "API authentication failed. Check your GOOGLE_API_KEY in the .env file."
- If the API is temporarily unavailable (503 Service Unavailable), Django retries the request 3 times with exponential backoff (wait 1 second, then 2, then 4). If all retries fail, the task fails, and the user sees: "Gemini API is temporarily unavailable. Please try again in a few minutes."
- If the prompt violates Google's content policy (e.g., explicitly violent content), Google returns a 400 error with a message like "Prompt blocked by safety filters." Django passes this to the user: "Video generation failed: Your prompt was blocked for policy violations."
- If the request times out (no response after 5 minutes), Django cancels the task and notifies the user: "Generation timed out. Please try again with a shorter prompt."

**What happens if this connection is slow or fails**: Video generation already takes minutes, so slow API responses are expected. If the internet connection is very slow, uploads (sending reference images) take longer, but you only notice a longer generation time. If the connection fails mid-request, the thread retries (as described above). If the connection is completely down, all generations fail immediately with a network error.

### Background Threads (Django ↔ Database)

**How they talk**: When Django spawns a background thread for a long-running task, the thread uses the same database connection as Django's main process. The thread creates a Task record (via Django ORM: `Task.objects.create(status='running', task_id=unique_id)`), performs the work (AI API call, file download), and updates the Task record when done (`task.status = 'completed'; task.save()`). The frontend polls a Django view that queries the Tasks table (`Task.objects.get(task_id=abc123)`) and returns the status as JSON.

**What format the data travels in**: Python objects (Django ORM models) in memory. When saved, they become SQL queries and database rows.

**How errors at this boundary are handled**:
- If a thread tries to write to the database while another thread is writing, SQLite queues the write (brief delay, milliseconds). No error unless the timeout expires (rare).
- If a thread crashes before updating the task status, the task record stays "running" forever. The frontend polls indefinitely. You must manually retry or run a cleanup script to mark stale tasks as "failed."
- If the thread raises an unhandled exception, Python logs it to stderr (the terminal running `runserver`). The thread exits without updating the database. The task appears stuck.

**What happens if this connection is slow or fails**: SQLite is file-based, so writes are as fast as the hard drive. SSDs make this nearly instant (milliseconds). If the database file is locked (another thread is writing), the ORM waits (default timeout: 5 seconds). If the timeout expires, Django raises an exception. For a solo user, this is extremely rare.

## 8. Security — How the System Stays Safe

This section explains how Peliku protects itself and your data, in beginner-friendly terms.

### How User Passwords Are Handled

Peliku has **no user accounts**, so there are no passwords to handle. If you were to add authentication later, here is how it would work: "We never store your actual password. When you create an account, we run your password through a one-way mathematical function called a hash (like a blender — you put vegetables in and get soup out; you cannot reverse it to get the vegetables back). We store the soup (hashed password). When you log in, we hash what you typed and compare it to the stored soup. If they match, you are authenticated. Even if someone steals our database, they only get unreadable hashes, not your passwords."

### How the System Prevents Common Attacks

**XSS (Cross-Site Scripting)**: An attack where a malicious user injects JavaScript into your web page (e.g., by typing `<script>alert('hacked')</script>` into a form field). Django automatically escapes all user input in templates — it converts `<` to `&lt;` and `>` to `&gt;`, so the browser displays it as text instead of running it as code. You cannot XSS yourself in Peliku because you are the only user, but this protection exists.

**CSRF (Cross-Site Request Forgery)**: An attack where a malicious website tricks your browser into sending requests to Peliku without your knowledge. Django's CSRF protection is disabled in Peliku because there are no user sessions to hijack (no authentication). If authentication is added later, CSRF tokens (random secret codes) will be included in every form to ensure requests come from legitimate pages.

**SQL Injection**: An attack where a malicious user types SQL code into a form field to manipulate the database (e.g., typing `'; DROP TABLE projects; --` to delete all projects). Django's ORM parameterizes all queries — user input is never directly interpolated into SQL strings. Instead, it is passed as a parameter (like a placeholder), which the database sanitizes automatically. This makes SQL injection impossible.

### How API Endpoints Are Protected from Abuse (Rate Limiting)

Peliku has **no rate limiting** because there is only one user (you). You cannot abuse your own app. If the app is ever exposed to the internet, rate limiting would be added via a Django middleware or nginx: "Allow 10 video generation requests per minute per IP address. Block the IP for 5 minutes if the limit is exceeded." This prevents someone from spamming the API and running up your Gemini API bill.

### How Secrets (API Keys, Database Passwords) Are Managed

The Gemini API key is stored in a `.env` file (a plain text file with key-value pairs: `GOOGLE_API_KEY=your-key-here`). This file is:
1. **Gitignored** (never committed to version control, so it does not appear on GitHub or GitLab if you push your code).
2. **Loaded at runtime** by Django using a library like `python-decouple`. Django reads the key from the `.env` file and stores it in memory. The key is never hardcoded in the source code.
3. **Protected by file permissions**: In production, the `.env` file has restricted permissions (only the user running Django can read it).

Why this matters: "If your API key is in your source code and you push your code to GitHub, anyone can see it and use your key, running up your bill. By keeping it in a `.env` file and gitignoring that file, only you have the key."

### How HTTPS Works and Why It Matters

If you run Peliku on your local machine (`http://localhost:8000`), all traffic stays inside your computer — no encryption needed. If you deploy to a server with a domain (e.g., `https://peliku.yourserver.com`), HTTPS encrypts all communication between your browser and the server using a protocol called TLS (Transport Layer Security). Here is how it works:
1. Your browser connects to the server and asks for its **SSL certificate** (a digital ID proving the server is who it claims to be).
2. The server sends the certificate.
3. Your browser verifies the certificate is valid (signed by a trusted authority, not expired).
4. Your browser and server negotiate an encryption key (a secret code known only to them).
5. All subsequent traffic (requests, responses, cookies) is encrypted using that key. Anyone intercepting the traffic (on public Wi-Fi, for example) sees gibberish.

Why it matters: "Without HTTPS, anyone on the same network can see everything you send to Peliku: your project titles, scripts, API key (if accidentally sent in a request). With HTTPS, it is all encrypted."

## 9. Deployment — How Code Gets to Users

For Peliku, "users" means you — the single person using the tool. Deployment is the process of setting up the app so you can use it, either on your own computer or a remote server.

### The Deployment Pipeline

Here is the journey from development to running app:

1. **A developer writes code and pushes it to a Git repository**: You write code on your machine (editing Python files, CSS files, templates) and commit it to a Git repository. You push it to a remote repository (GitHub, GitLab, or self-hosted) so you can pull it on your other development machine.

2. **No automated testing**: For a solo dev tool, there is no CI/CD pipeline. You manually run tests when you remember: `pytest` (runs unit tests) and `black . && flake8 . && isort .` (formats and lints code). If tests pass and code is clean, you proceed.

3. **The code runs on your development machine**: You run `python manage.py runserver` on your computer. The app is accessible at `http://127.0.0.1:8000` or `http://localhost:8000`. This is the only deployment — there is no production server, no staging environment, no cloud hosting. Just your dev machine.

4. **You see the app**: Open your browser to localhost and use the app. When you push code from one dev machine, you pull it on the other and run the server there.

### Environments

**Development**: Your local machine. You run Django's dev server (`python manage.py runserver`), which auto-reloads when you change code. Debug mode is enabled (detailed error pages). You use SQLite (the database file is in the project root). The `.env` file has your Gemini API key.

**No staging or production**: This tool only runs on your development machines. No deployment to servers, no public access, no SSL certificates, no production configurations.

### Rollbacks

If you write new code and something breaks:
1. **Git rollback**: `git revert <commit-hash>` or `git reset --hard <previous-commit>`. This reverts the code to the last working version.
2. **Restart the server**: Stop `runserver` (Ctrl+C) and start it again.
3. **Database rollback**: If the broken code included a database migration that corrupted data, restore the database from a backup (copy a backup `db.sqlite3` over the live one, or run `python manage.py migrate <app_name> <previous_migration_number>` to roll back migrations).

## 10. What Happens When Things Go Wrong

This section explains how Peliku detects, handles, and recovers from failures.

### How Errors Are Detected and Who Gets Alerted

**In development**: Errors appear immediately in the browser (Django's debug page with full traceback — a detailed stack trace showing exactly which line of code failed and why). The terminal running `runserver` also shows the error.

**Who gets alerted**: You (the developer/user) see the error immediately when you use the app. No email alerts, no monitoring services. Just error pages and terminal logs.

### How You Know If the App Is Slow or Down

**In development**: You notice immediately because you are using it. If the server is down (you stopped `runserver`), the browser shows "connection refused." If the app is slow, you notice actions taking longer than usual.

### What Happens When an External Service Fails

**Gemini API is down**: All generations fail. You see error messages: "Video generation failed: API unavailable." The scripts and existing videos are unaffected. You wait and retry. No data loss.

**Hard drive fills up**: Video generation fails ("disk full" error). You delete old projects or videos to free space and retry.

### How the System Recovers from Common Failure Scenarios

**Scenario: Background thread crashes mid-generation**
- Detection: After 10 minutes of no progress, you notice the status is stuck on "running." The task record in the database is never updated to "completed."
- User impact: The frontend keeps polling forever (or until timeout). You see "Generating..." forever.
- Recovery: You refresh the page and click "Generate Video" again to retry. A cleanup script could detect stale "running" tasks (started more than 10 minutes ago) and mark them as "failed" on server restart.

**Scenario: Django server crashes**
- Detection: Browser shows "connection refused." You notice immediately.
- Recovery: You restart Django (`python manage.py runserver` in the terminal). All data is safe (SQLite file and media files are untouched). Any in-flight threads are lost — you retry those operations.

**Scenario: Power outage during video generation**
- Detection: When your machine restarts, the Django server is not running. The Tasks table has a stale "running" entry.
- User impact: The clip's status stays "generating" in the database. You notice it never completes.
- Recovery: You manually regenerate the clip. A cleanup script could detect stale "running" statuses and reset them to "idle" on startup.

## 11. Key Design Decisions — Why This and Not That

This section explains the major architectural choices and why they fit this specific product.

### Decision: Monolithic Architecture (Django Monolith)

**What was chosen**: A single Django application does everything (UI, API, database access, file storage, AI coordination).

**The alternatives**: Microservices (separate services for the frontend, backend, AI coordination, video processing) or serverless (AWS Lambda functions for each operation).

**The reasoning**: Peliku is a single-user tool with straightforward requirements. A monolith minimizes complexity: one codebase, one deployment, one server, one database. Debugging is easy (all logs in one place). No inter-service communication overhead. Microservices make sense for large teams working on different services independently or for apps that need to scale individual components independently (e.g., scale the video processing service to 100 servers while keeping the UI on one server). Here, there is one server, one user, and no team. A monolith is the simplest solution.

**The trade-off**: If Peliku ever becomes multi-user with millions of users, a monolith is harder to scale horizontally (you would need to run multiple identical servers behind a load balancer and share the database and file storage). But the specs explicitly exclude multi-user scenarios. For V1, simplicity wins.

### Decision: SQLite (Relational, File-Based Database)

**What was chosen**: SQLite, a self-contained, file-based relational database.

**The alternatives**: PostgreSQL (client-server relational database), MongoDB (NoSQL document database), or Firebase (cloud-hosted NoSQL).

**The reasoning**: Peliku's data is highly relational: projects have clips, clips have videos, projects have reference images. Relational databases excel at modeling these relationships. SQLite is the simplest relational database: zero configuration, no separate server process, perfect for a solo tool. PostgreSQL would add installation complexity and resource usage (database server running in the background) for no benefit at this scale. MongoDB would require restructuring the data model to fit its document-based paradigm (projects as documents with nested clip arrays), which is more awkward for this use case.

**The trade-off**: SQLite serializes writes (only one write at a time). In a multi-user app with concurrent edits, this is a bottleneck. But with one user, writes never overlap. SQLite is also limited to the size of the file system (realistically, databases larger than 1GB start slowing down). For a tool with hundreds of projects, this is well within limits. If the database file gets corrupted (rare, but possible), you restore from a backup.

### Decision: Local File System Storage

**What was chosen**: Video files and images are stored on the local hard drive in a `media/` folder.

**The alternatives**: Cloud object storage (AWS S3, Google Cloud Storage, Cloudflare R2).

**The reasoning**: The user explicitly prefers local storage. It is simpler (no SDK, no credentials, no API calls, no monthly bills), faster (no upload/download time — files are already on the machine), and free. Cloud storage makes sense for multi-user apps where all users need to access the same files or for apps deployed across multiple servers (shared storage layer). Here, one server, one user, one file system.

**The trade-off**: No built-in redundancy. If the hard drive fails, all videos are lost. The user must handle backups manually (copy `media/` and `db.sqlite3` to an external drive or cloud backup service). Acceptable for a personal tool.

### Decision: Server-Side Rendering (Django Templates)

**What was chosen**: Django generates complete HTML pages on the server and sends them to the browser. JavaScript enhances interactivity via AJAX.

**The alternatives**: Single-page application (SPA) with React/Vue (server sends a minimal HTML shell, JavaScript builds the UI dynamically) or static site generation (SSG) with Next.js/Gatsby (pre-render all pages at build time).

**The reasoning**: SSR is simpler: no build step, no client-side router, no state management library, no hydration bugs. Pages load fast (HTML is ready when it arrives). The specs do not require instant client-side navigation or complex UI state — the user spends most of their time in the Project Workspace looking at one project. SPAs shine in apps with frequent navigation and dynamic updates (dashboards, admin panels, social media feeds). Here, the bottleneck is AI generation (minutes), not page load time (milliseconds). SSR fits perfectly.

**The trade-off**: Full page reloads when navigating (home page → workspace → home page). SPAs would avoid reloads (instant transitions). But the user navigates infrequently (they open a project and stay there for 10 minutes). The reload delay (< 100ms) is imperceptible.

## 12. How the System Grows

This section explains what happens as usage increases — not user count (out of scope), but data volume and frequency of use.

### What Parts of the System Will Feel Pressure First

**Storage** is the first bottleneck. High-resolution videos are large (100-500MB per clip). A project with 10 extended 4K clips is 5GB. If the user creates 20 such projects, that is 100GB. The hard drive eventually fills up. Mitigation: the user deletes old projects or adds more storage.

**Gemini API costs** are the second pressure point. Video generation via Veo is pay-per-use and expected to be expensive (pricing TBD, but likely $0.50–$2 per video). If the user generates 50 clips per week, monthly costs could reach $100–$400. Mitigation: use the "Fast" generation mode for drafts, reserve "Quality" for final renders.

### What the Plan Is When the Database Gets Large

At 1000 projects with 7 clips each (7000 clips), the database file might be 50-100MB (metadata only, not videos). SQLite handles this easily. At 10,000 projects, the file could be 500MB–1GB, and queries might slow down. Mitigation:
- Add indexes to frequently queried columns (`projects.updated_at`, `clips.project_id`).
- Archive old projects (move them to a separate "archive" database file).
- If performance degrades significantly, migrate to PostgreSQL (Django's ORM makes this straightforward — change the database setting, export data, import to PostgreSQL).

### What the Plan Is When Traffic Spikes

Not applicable. One user cannot spike traffic.

### What Architectural Changes Might Be Needed at 10x or 100x Scale

**10x scale (10 users)**: Add authentication (user accounts), deploy to a server accessible over the internet, switch to PostgreSQL (for concurrent writes), add S3 storage (shared file storage), add horizontal scaling (multiple Django servers behind a load balancer). The architecture stays mostly the same — just more users.

**100x scale (100 users)**: Separate video generation into a dedicated service (queue jobs in a database or RabbitMQ instead of Redis, run Celery workers on separate machines), add a CDN (Cloudflare) to cache video files, add database read replicas (for faster reads), add caching (Redis for session data and query results), add real-time status updates via WebSockets (instead of polling). The architecture shifts toward microservices.

These are thought exercises — V1 is a solo tool, and these changes would cost weeks of engineering time for zero current benefit.

### What Was Intentionally Kept Simple for V1 That Will Need Revisiting Later

**No version history**: If you regenerate a clip's video, the old video is deleted. A versioning system would save all previous versions and allow rollback. Excluded for V1 to save storage and complexity. Could be added later using a `ClipVersion` table.

**No parallel batch generation**: You must generate each clip individually. Excluded for V1 because the user reviews each clip before moving to the next. Could be added later as a Celery group task (enqueue 7 jobs at once).

**No custom transition effects**: Transitions are handled by frame interpolation (smooth content-level transitions) but no effects like crossfade or wipe. Excluded to keep scope focused. Could be added later using FFmpeg's `xfade` filter during assembly.

**No cloud deployment guide**: Deployment instructions assume a self-hosted server or local machine. If the user wants to deploy to AWS, DigitalOcean, or Heroku, they must figure it out themselves. A V2 deployment guide could cover this.

## 13. Glossary

**AJAX**: Asynchronous JavaScript and XML. A technique where JavaScript sends requests to the server in the background (without reloading the page) and updates the UI when the response arrives.

**API**: Application Programming Interface. A set of endpoints (URLs) a server exposes so programs can interact with it. Example: `/api/clips/3/generate` is an API endpoint.

**API key**: A secret password-like code you send with API requests to prove you have permission to use the service. The Gemini API key is stored in the `.env` file.

**Background thread**: A lightweight subprocess within your Django server that runs long-running tasks (like video generation) without blocking the main server. Python's `threading` module creates threads.

**Base64**: A way of encoding binary data (like images) as text so it can be sent in JSON. Used when uploading reference images to the Gemini API.

**CDN**: Content Delivery Network. A network of servers worldwide that cache and serve static files (images, videos, CSS) to users faster than a single origin server could. Not used in Peliku.

**Client**: Your browser. In web architecture, the client is the program requesting data (the browser), and the server is the program providing data (Django).

**CSS**: Cascading Style Sheets. The language that controls visual styling on web pages (colors, fonts, spacing, layout).

**Database**: A program or file that stores structured data (projects, clips, settings). Peliku uses SQLite.

**Deployment**: The process of setting up the app so it can be used — either on your local machine (development) or a server (production).

**Django**: A Python web framework. It handles HTTP requests, database access, HTML rendering, and file uploads. The backbone of Peliku.

**Endpoint**: A specific URL on a server that performs an action. Example: `POST /api/clips/3/generate` is an endpoint for generating a video.

**Environment**: A context where the app runs. "Development" is your local machine for testing; "production" is the real server where you use the app.

**FFmpeg**: A command-line tool for processing video (cutting, concatenating, converting formats, adding effects). Used to assemble the final reel in Peliku.

**Foreign key**: A database column that links two tables. Example: the `Clips` table has a `project_id` column (a foreign key) that links each clip to its project.

**Gemini**: Google's family of AI models. Includes text (Gemini 3), image (Nano Banana), and video (Veo) generation.

**HTML**: HyperText Markup Language. The language browsers use to structure web pages (headings, paragraphs, buttons, forms).

**HTTP**: HyperText Transfer Protocol. The language browsers and servers use to communicate. You send an HTTP request (GET /home), the server sends an HTTP response (the HTML for the home page).

**HTTPS**: HTTP Secure. HTTP with encryption (TLS) so traffic cannot be intercepted and read.

**JavaScript**: A programming language that runs in your browser. Used to make web pages interactive (responding to clicks, updating content without reloading).

**JSON**: JavaScript Object Notation. A text format for sending structured data. Example: `{ "clipId": 3, "status": "complete" }`.

**jQuery**: A JavaScript library that simplifies common tasks (DOM manipulation, AJAX, event handling). Used extensively in Peliku.

**Middleware**: A piece of code that runs on every HTTP request before reaching the view function. Used for logging, authentication, CSRF protection, etc. Not heavily used in Peliku since there is no authentication.

**Model**: In Django, a Python class representing a database table. Example: the `Project` model represents the `projects` table.

**MP4**: A video file container format. Most video players support MP4. Veo generates MP4 files.

**Nginx**: A web server and reverse proxy. In production, nginx receives HTTP requests, serves static files directly (images, CSS, JS), and forwards dynamic requests to Gunicorn (which runs Django).

**NoSQL**: A type of database that stores data as documents (like JSON) instead of tables. Example: MongoDB. Not used in Peliku.

**ORM**: Object-Relational Mapper. A library (built into Django) that lets you interact with the database using Python code instead of SQL. Example: `Project.objects.all()` retrieves all projects.

**Relational database**: A type of database that stores data in tables with rows and columns, and links tables using foreign keys. Example: SQLite, PostgreSQL.

**REST**: Representational State Transfer. A style of API design where each endpoint represents a resource (e.g., `/api/clips/3`) and HTTP methods (GET, POST, PUT, DELETE) represent actions on that resource.

**SDK**: Software Development Kit. A library that simplifies using an API. The `google-genai` Python SDK makes it easy to call the Gemini API without manually constructing HTTP requests.

**Server**: The computer (or program) that responds to requests from clients (browsers). Django is the server program in Peliku.

**SQL**: Structured Query Language. The language relational databases use to read and write data. Example: `SELECT * FROM projects WHERE id = 5`.

**SQLite**: A lightweight, file-based relational database. The entire database is a single file (`db.sqlite3`). Used in Peliku.

**SSL**: Secure Sockets Layer. The old name for TLS (the protocol that encrypts HTTPS). Often used interchangeably with TLS.

**Template**: An HTML file with placeholders for dynamic content. Django fills in the placeholders with data from the database and sends the complete HTML to the browser.

**TLS**: Transport Layer Security. The protocol that encrypts internet traffic (used in HTTPS). Prevents eavesdropping.

**Token**: A random string used as a temporary access key. In authentication systems, the server gives your browser a token after you log in, and your browser includes the token in every request. Peliku does not use tokens (no authentication).

**View**: In Django, a Python function that handles an HTTP request. Example: `def home(request):` is the view function for the home page.

**Webhook**: A way for one server to notify another server when something happens (by sending an HTTP POST request). Not used in Peliku — the browser polls instead.

**WSGI**: Web Server Gateway Interface. A standard way for web servers (like Gunicorn or nginx) to communicate with Python web apps (like Django).
